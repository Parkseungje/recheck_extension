# 📖 ReCheck — 기획서

> **작업명(가칭): ReCheck** *"읽었다"와 "이해했다" 사이를 다시 확인시키는 크롬 확장*
> 문서 버전: v0.2 (초안) · 목적: 개인 학습용 + 포트폴리오
> v0.2 변경: 기술 글 한정 → **범용 4모드**(적용/검증/논증/소화) 구조로 확장. 프롬프트를 "공통 뼈대 + 모드 조각"으로 재편.

---

## 1. 한 줄 정의

읽은 글에 대해 사용자가 **스스로 자신의 이해를 다시 돌아보게** 만드는 크롬 확장. AI는 정답을 주지 않는다. **질문과 힌트만** 던지고, 답에 도달하는 것은 사용자의 몫이다.

---

## 2. 문제 정의 — 왜 만드는가

글이 매끄럽게 읽히면 뇌는 "안다"고 착각한다(**이해 착각, illusion of comprehension**). 하지만 막상 설명하거나 응용하거나 비판하려 하면 막힌다. 이 간극은 기술 문서뿐 아니라 뉴스·칼럼·에세이 등 **모든 종류의 글**에서 나타난다. 다만 글의 종류에 따라 "제대로 이해했는지"를 확인하는 방식은 달라야 한다 (→ 5장 4모드).

기존 도구(ChatGPT, Claude 등 챗봇)는 물어보면 **답을 준다.** 답을 받는 순간 사용자는 "그렇구나" 하고 닫으며, 이해 착각은 그대로 남는다. 회수(retrieval)가 일어나지 않으므로 학습으로 남지 않는다.

**ReCheck는 정답을 주지 않음으로써 이 문제를 푼다.**

---

## 3. 핵심 차별점 — 챗봇과 무엇이 다른가

| | 챗봇 (GPT/Claude) | ReCheck |
|---|---|---|
| AI의 역할 | 답을 준다 | 질문·힌트만 던진다 |
| 사용자 행동 | 읽고 수용 (수동) | 회수·적용 (능동) |
| 이해 착각 | 강화됨 | 파괴됨 |
| 최종 종착지 | AI의 답 | **원문으로 복귀** |
| 오개념 위험 | AI가 틀리면 주입됨 | **AI가 사실을 단정하지 않아 0** |

> **설계 대원칙**: AI는 "심판"이 아니라 "사다리"다. 정답 단어를 발화하지 않고, 사용자가 스스로 그 단어에 도달하도록 계단만 놓는다.

---

## 4. 설계 원칙 (변경 불가 핵심 규칙)

1. **정답 미제공** — AI는 어떤 질문에도 정답을 생성하지 않는다. 사실을 단정하는 문장을 출력하지 않는다.
2. **유저 호출 시에만 작동** — 자동 팝업 금지. 사용자가 사이드패널을 열고 버튼을 눌러야 질문이 나온다. (읽기 흐름을 끊는 개입은 제품을 죽인다.)
3. **전이(transfer) 질문 중심** — 글 내용을 그대로 되묻지 않는다. "글이 멈춘 바로 다음 한 칸"을 묻는다.
4. **힌트는 정답 직전에서 멈춘다** — 3단계로 점점 좁아지되, 마지막 단계도 정답 단어를 말하지 않는다.
5. **막힘의 종착지는 원문** — 힌트를 다 써도 정답 대신 "원문의 이 문단을 다시 읽어라"로 사용자를 소스로 돌려보낸다. (grounded 안전장치)
6. **성찰은 사용자 몫** — 답 작성 후 AI 평가 없이 "네가 쓴 답을 다시 읽어봐"로 마무리한다.

---

## 5. 질문 생성 전략 — 4모드

### 5.1 왜 "장르"가 아니라 "독자 목적"으로 나누는가

글 유형을 겉모습(장르: 뉴스·칼럼·논문·레시피·여행기…)으로 나누면 수십 개로 늘어나고 확장 불가능해진다. 대신 **"독자가 이 글을 왜 읽었는가"**로 나누면 4개로 수렴한다. 장르가 아무리 늘어도 새 글은 이 4개 중 하나에 떨어진다.

| 모드 | 핵심 질문 | 이 방식이 먹히는 글 | 독자 목적 |
|---|---|---|---|
| **① 적용 (Transfer)** | "조건이 바뀌면?" | 기술·개념·튜토리얼·방법론·레시피 | 써먹으려고 |
| **② 검증 (Scrutiny)** | "사실이면 함의는? 안 다뤄진 면은?" | 뉴스·보도·발표 | 무슨 일인지 알려고 |
| **③ 논증 (Argument)** | "근거는? 반론은?" | 칼럼·사설·주장 글 | 누가 뭘 주장하는지 보려고 |
| **④ 소화 (Digest)** | "네 말로 요약하면? 네 경험과 어떻게 닿아?" | 에세이·교양·서사 | 그냥 읽고 싶어서 |

### 5.2 모드별 질문 공식

**① 적용 — "주장 → 성립 조건 → 조건 붕괴"**
글의 핵심 개념이 성립하는 숨은 조건을 찾고, 그 조건이 깨지는 상황을 묻는다.
> 예 (JPA): "1차 캐시로 `m1==m2`" → 조건 "트랜잭션 안" → "트랜잭션 **밖**으로 넘겨 지연 로딩에 접근하면?"

**② 검증 — "사실 → 함의 → 사각지대"**
핵심 사실 주장을 고르고, 그 함의(누구에게 영향이 가나)와 기사가 다루지 않은 반대편·이해당사자를 묻는다.
> 예 (금리 인상 기사): "이 인상으로 이득 보는 쪽과 손해 보는 쪽은? 기사가 언급하지 않은 집단은 누구일까?"

**③ 논증 — "주장 → 근거 → 반론"**
글쓴이의 핵심 주장과 그 근거를 짚게 하고, 가장 강력한 반론을 스스로 떠올리게 묻는다.
> 예 (사설): "글쓴이 주장의 근거는 무엇이었어? 이 주장에 가장 세게 반대하는 사람은 뭐라고 할까?"

**④ 소화 — "재구성 → 연결"**
핵심을 독자 자신의 말로 다시 구성하게 하고, 독자 경험·기존 지식과의 연결을 묻는다.
> 예 (에세이): "방금 읽은 걸 네 말로 세 줄로 옮기면? 네 경험 중 이 글과 닿는 순간이 있었어?"

### 5.3 공통 원칙

네 모드 모두 **정답을 주지 않는다.** 질문 방식만 다를 뿐, "AI는 사다리일 뿐 심판이 아니다"라는 대원칙은 동일하다. 어떤 모드든 글의 난이도가 아니라 "글이 멈춘 다음 칸 / 글이 안 다룬 면 / 독자 자신의 사고"를 건드리므로, 쉬운 글에서도 질문이 유치해지지 않는다.

## 6. 힌트 사다리 설계

각 질문마다 **3단 힌트 + 원문 포인터**를 붙인다. 힌트는 가려진 상태로 나오고, 클릭하면 하나씩 열린다.

| 단계 | 성격 | 예시 (더티 체킹 질문) |
|---|---|---|
| 힌트 1 | 개념 방향만 | "더티 체킹은 '뭐가 바뀌었나'를 어떻게 알아낼까?" |
| 힌트 2 | 사고의 틀 좁힘 | "매번 정확히 추려 쿼리를 새로 짜기 vs 항상 같은 쿼리 재사용, 뭐가 쌀까?" |
| 힌트 3 | 정답 직전 (단어는 안 말함) | "이 기본 동작을 바꾸는 스위치가 있어. 이름에 'Dynamic'이 들어가." |
| 원문 포인터 | 최종 안전장치 | "그래도 막히면 원문의 '쓰기 지연' 문단을 다시 읽어봐." |

> **힌트를 가려서 클릭으로 여는 UX는 '힌트'에만 적용한다. 정답에는 절대 적용하지 않는다.** (답을 가려두면 사용자는 무조건 먼저 열고, 그 순간 회수가 죽는다.)

## 7. 사용자 플로우

```
① 글 읽는 중 사이드패널 열기
② "질문 생성" 버튼 클릭
③ Readability로 본문 추출 → 선택한 provider API 호출 (1회)
   (이 1회 안에서 모드 판별 + 질문 생성이 함께 이뤄짐)
④ 판별된 모드 표시 + 질문 2개 (각 3단 힌트 가려진 상태 + 원문 포인터)
   (모드가 틀렸으면 드롭다운으로 바꿔 재생성 가능)
⑤ 사용자가 답 작성 (막히면 힌트를 하나씩 열어봄)
⑥ "네가 방금 쓴 답을 다시 읽어봐" → 성찰 마무리
⑦ 끝 (저장 없음)
```

AI가 등장하는 지점은 **③ 질문 생성 딱 한 번.** 이후는 전부 사용자 혼자.

## 8. 기술 아키텍처

### 8.1 전체 구조 (서버 없음)

```
[크롬 확장 — Manifest V3]
 ├─ content script : Readability.js로 현재 페이지 본문 추출 (EXTRACT 메시지 수신 시에만)
 ├─ side panel (UI) : 질문/힌트 렌더 + provider API 직접 fetch
 ├─ service worker  : 아이콘 클릭 시 사이드패널 오픈 설정
 └─ chrome.storage.local : provider 선택값 · 모델 · API 키 · 프롬프트 뼈대 저장
```

> API 호출은 service worker가 아니라 **사이드패널(확장 페이지)에서 직접 fetch**한다. MV3에서 확장 페이지의 fetch도 `host_permissions` 등록 도메인에 한해 CORS가 열리므로, 별도 메시지 중계 없이 UI에서 바로 호출하는 편이 단순하다. service worker는 패널 오픈 설정만 담당한다.

**서버 0 · 고정비 0.** 각 사용자가 자기 API 키를 넣고(BYOK), 자기 토큰 비용만 부담. 개인용·포트폴리오 목적에 최적.

### 8.2 멀티 Provider 추상화 (전략 패턴)

세 provider(GPT / Gemini / Claude)를 선택 가능. 차이를 adapter 한 곳에 가두고, 나머지 코드는 provider를 모른다. 각 adapter는 응답을 **"질문+힌트 JSON 하나"로 정규화**해서 반환한다.

| Provider | 엔드포인트 | 키 위치 | 특이사항 | 응답 경로 |
|---|---|---|---|---|
| Claude | `/v1/messages` | `x-api-key` 헤더 | `anthropic-dangerous-direct-browser-access: true` 헤더 필수 | `content[0].text` |
| GPT | `/v1/chat/completions` | `Authorization: Bearer` | 없음 | `choices[0].message.content` |
| Gemini | `/v1beta/models/*:generateContent` | URL `?key=` | 키를 URL에 노출 (로그 금지) | `candidates[0].content.parts[0].text` |

### 8.3 manifest.json 필수 설정

MV3 확장은 CORS 규칙이 웹페이지와 다르다. `host_permissions`에 세 API 도메인을 등록해야 service worker에서 외부 호출이 가능하다.

```json
"host_permissions": [
  "https://api.anthropic.com/*",
  "https://api.openai.com/*",
  "https://generativelanguage.googleapis.com/*"
]
```

### 8.4 기술 스택

| 영역 | 선택 | 이유 |
|---|---|---|
| 플랫폼 | Chrome Extension **Manifest V3** | 크롬 확장 현행 표준 |
| UI 표면 | **Chrome Side Panel API** | 글과 나란히 두고 답을 쓸 수 있음 (팝업 ❌) |
| 프레임워크 | **Vue 3 + Vite** (+ `@crxjs/vite-plugin`) | 본인 강점 활용 + 포트폴리오 |
| 언어 | TypeScript | provider adapter 타입 안전성 |
| 본문 추출 | **`@mozilla/readability`** | 광고·사이드바 제거하고 본문만 추출 |
| 저장소 | **`chrome.storage.local`** | provider 선택값·API 키·프롬프트 저장 (서버 불필요) |
| AI 호출 | 각 provider REST API **직접 fetch** | 서버·프록시 없음 (BYOK) |
| 백엔드 | **없음** | 유지비용 0의 핵심 |

> **프레임워크 대안**: 극단적 단순함을 원하면 **Vanilla JS + HTML/CSS**로 빌드 스텝 없이 가는 것도 가능(의존성·유지보수 최소). 다만 Vue 3는 본인 주력 스택이라 포트폴리오 가독성·개발 속도 면에서 유리. 확장 자체가 작으므로 어느 쪽도 무겁지 않음. **권장: Vue 3 + Vite** (빌드 스텝은 '실행 비용'이 아니라 '개발 편의'이므로 유지비용과 무관).

### 8.5 유지비용 (0을 목표)

| 항목 | 비용 | 비고 |
|---|---|---|
| 서버/호스팅 | **0원** | 백엔드 없음. 확장은 정적 파일로 로컬 로드 또는 스토어 배포 |
| DB | **0원** | 저장은 `chrome.storage.local` (사용자 기기) |
| 도메인 | **0원** | 필요 없음 |
| AI 토큰 | **각 사용자 부담 (BYOK)** | 본인이 쓰는 만큼만 각 provider에 과금 |
| 스토어 등록 | 초기 0원 | 개발자 모드 로컬 로드. (웹스토어 배포 시 1회성 $5 등록비 — 나중 선택) |

> **유지비용을 0으로 만드는 단 하나의 설계 결정 = "서버를 두지 않는다".** 서버를 두는 순간 고정비·결제·남용 방어가 딸려온다. BYOK는 수익화를 포기하는 대신 이 비용 구조를 성립시킨다. 포트폴리오 목적에는 이 교환이 옳다.

## 9. 프롬프트 설계 (⚠️ 외부화 — 나중에 수정 가능)

**프롬프트는 이 제품의 90%이자, 가장 자주 튜닝할 부분이다. 코드에 하드코딩하지 않는다.**

### 9.1 외부화 방식

- 시스템 프롬프트를 **별도 설정값**으로 관리 (`chrome.storage.local` 또는 확장 내 편집 가능한 설정 화면).
- 설정 UI에서 프롬프트 원문을 직접 열람·수정할 수 있게 한다.
- 기본 프롬프트(default)를 제공하되, 사용자가 덮어쓰거나 "기본값 복원" 가능.
- 이렇게 하면 재빌드 없이 프롬프트만 바꿔가며 질문 품질을 실험할 수 있다.

### 9.2 출력 JSON 스키마 (확정 초안)

```json
{
  "mode": "적용 | 검증 | 논증 | 소화",
  "questions": [
    {
      "question": "질문 텍스트 (판별된 모드의 공식에 따름)",
      "hints": [
        "힌트1: 방향만",
        "힌트2: 사고의 틀 좁힘",
        "힌트3: 정답 직전 (정답 단어는 말하지 않음)"
      ],
      "source_pointer": "원문에서 다시 읽을 부분 안내 (정답 아님)"
    }
  ]
}
```

- `mode`: 모델이 판별한 글 유형(4모드 중 하나). UI에 표시하고, 틀리면 사용자가 바꿔 재생성.
- `questions` 배열 길이는 항상 2.
- `hints` 배열 길이는 항상 3.
- adapter들은 각 provider 응답을 이 스키마로 정규화하는 것을 목표로 한다.

### 9.3 프롬프트 구조 — 공통 뼈대 1 + 모드 조각 4

프롬프트를 모드별로 통째로 4개 만들지 않는다. **공통 뼈대 하나**에 **판별된 모드의 조각 하나**를 끼워 조립한다. 이렇게 하면 안전 규칙(정답 금지 등)이 한 곳에만 있어 유지가 쉽고, 새 모드 추가도 조각 하나만 쓰면 된다.

```
최종 프롬프트 = 공통 뼈대  +  [모드 판별]  +  선택된 모드 조각
```

#### 9.3.1 공통 뼈대 (기본값 — 항상 동일)

> 설정 화면에서 열람·수정하고 "기본값 복원" 가능하게 한다.

```text
너는 사용자가 방금 읽은 글에 대해 "진짜 이해했는지" 스스로 돌아보게
만드는 질문 생성기다. 너의 역할은 오직 질문과 힌트를 던지는 것이며,
너는 절대로 답이나 설명, 사실 주장을 제공하지 않는다.

# 입력
사용자가 방금 읽은 글의 본문 텍스트가 주어진다.

# 1단계 — 모드 판별
먼저 이 글을 읽은 독자의 목적이 아래 4가지 중 무엇에 가장 가까운지
판단하라. 판단한 모드를 출력 JSON의 mode 필드에 넣어라.
- 적용: 배워서 써먹는 글 (기술·개념·튜토리얼·방법론·레시피)
- 검증: 사실을 전달하는 글 (뉴스·보도·발표)
- 논증: 설득하려는 글 (칼럼·사설·주장)
- 소화: 읽고 음미하는 글 (에세이·교양·서사)

# 2단계 — 질문 생성
판별한 모드의 [모드 공식]에 따라 질문 2개를 만든다.
질문은 글 내용을 그대로 되묻지 않고, 독자가 스스로 사고하게 만든다.

## 힌트 (각 질문마다 정확히 3단계)
- 힌트1: 무엇을 떠올려야 하는지 "방향"만 가리킨다.
- 힌트2: 사고의 틀을 한 번 더 좁힌다.
- 힌트3: 정답 바로 앞까지 데려간다. 단, 정답에 해당하는
        핵심 단어·용어·이름은 절대 직접 말하지 않는다.

## source_pointer (각 질문마다)
- 힌트를 다 써도 막힐 때를 대비해 원문에서 다시 볼 부분을 안내한다.
  이것은 "정답"이 아니라 "어디를 보라"는 포인터다.

# 절대 규칙 (위반 시 실패)
- 정답을 말하지 마라. 어떤 사실도 단정하지 마라.
- 모든 문장은 "질문" 또는 "어디를 보라"는 포인터 형태여야 한다.
  ("~이다", "~하면 ~가 된다" 같은 단정형 서술 금지.)
- 정답에 해당하는 핵심 용어를 힌트에 직접 쓰지 마라.
- 글을 요약하지 마라.
- 질문 정확히 2개, 각 힌트 정확히 3개.
- 글과 같은 언어로 작성하라. (한글 글이면 한글, 영어 글이면 영어.)

# 출력 형식
아래 JSON만 출력한다. 코드펜스(```), 설명, 인사말 등 그 외 텍스트를
일절 붙이지 마라.

{
  "mode": "적용 | 검증 | 논증 | 소화",
  "questions": [
    { "question": "...", "hints": ["...","...","..."], "source_pointer": "..." },
    { "question": "...", "hints": ["...","...","..."], "source_pointer": "..." }
  ]
}
```

#### 9.3.2 모드 조각 ([모드 공식] 자리에 끼워짐)

```text
[적용 모드 공식]
1. 글의 핵심 개념 하나를 고른다.
2. 그 개념이 성립하는 "숨은 조건"을 찾는다.
3. 그 "조건이 깨지는 상황"을 질문으로 만든다.
   (예: "트랜잭션 안에서 성립" → "트랜잭션 밖이면?")

[검증 모드 공식]
1. 글의 핵심 "사실 주장" 하나를 고른다.
2. 그 사실의 함의(누구에게 어떤 영향이 가는지)를 질문으로 만든다.
3. 글이 다루지 않은 반대편·이해당사자·빠진 맥락을 질문으로 만든다.

[논증 모드 공식]
1. 글쓴이의 핵심 "주장"과 그 "근거"를 짚게 하는 질문을 만든다.
2. 그 주장에 대한 가장 강력한 "반론"을 독자가 스스로
   떠올리게 하는 질문을 만든다.

[소화 모드 공식]
1. 글의 핵심을 독자 "자신의 말"로 재구성하게 하는 질문을 만든다.
2. 그 내용을 독자의 경험·기존 지식과 연결하게 하는 질문을 만든다.
```

> **재사용 참고**: 기존 기술 특화 프롬프트(구 9.3)는 버려지지 않았다. 그 안전 뼈대가 위 공통 뼈대로 올라갔고, 기술 특화 부분이 [적용 모드 공식]으로 내려온 것이다.

### 9.4 구현 단계별 프롬프트 적용

MVP에서는 **[적용 모드]부터 하나만** 붙여 완성한다. 4모드를 동시에 튜닝하면 어느 모드가 왜 별로인지 추적이 안 된다. 적용 모드로 접근이 통하는 것을 확인한 뒤, 검증→논증→소화 조각을 하나씩 붙인다. (뼈대+조각 구조라 조각 추가 비용이 싸다.)

### 9.5 프롬프트 튜닝 시 관찰할 것

실제로 돌려보며 다음 실패 모드를 관찰하고 문구를 조정한다:

- **정답 누설** — 힌트3에서 핵심 용어를 말해버리는가? (가장 흔한 실패)
- **모드 오판** — 글 유형을 엉뚱하게 판별하는가? (→ 사용자 수동 재선택으로 방어)
- **단정형 혼입** — "~이다" 같은 사실 주장이 섞이는가?
- **되묻기 함정** — 글을 그대로 되묻는 얕은 질문이 나오는가?
- **JSON 오염** — 코드펜스·설명이 앞뒤에 붙는가? (특히 Gemini. adapter에서 방어 필요)

## 10. 보안 고려사항

- API 키는 **반드시 `chrome.storage.local`에 저장.** `localStorage` 금지 (페이지 JS·악성 스크립트가 접근 가능).
- `chrome.storage.local`도 암호화 저장은 아님 — 확장 컨텍스트에 격리될 뿐. 개인용 로컬 설치라 현실 위험은 낮으나 인지할 것.
- **키를 코드에 하드코딩 금지.** (스토어 배포 시 키 탈취 위험 — 애초에 이 구조는 스토어 배포용이 아님.)
- Gemini는 키가 URL에 실리므로 **로그·에러 메시지·네트워크 캡처에 키 노출 금지.**
- Claude의 `dangerous-direct-browser-access` 헤더 이름이 경고하듯, 브라우저 직접 호출은 키 노출을 감수하는 구조. 개인용에 한정.

## 11. 범위 정의

### MVP에 포함 (In Scope)
- 본문 추출 (Readability.js)
- provider 선택 + 키 입력 설정 화면
- 질문 2개 + 3단 힌트 생성 및 렌더
- 답 입력창 + 성찰 마무리 문구
- 프롬프트 편집(외부화) 기능

### 범위 밖 (Out of Scope — 의도적으로 제외)
- ❌ 답변 저장 / 히스토리 (붙이는 순간 백엔드·DB 필요 → 나중에)
- ❌ 자동 팝업 / 스케줄 알림
- ❌ 메타인지 슬라이더 (1~5 자가진단) — 마찰만 늘어 제외
- ❌ AI의 답변 평가·채점
- ❌ 크롬 웹스토어 배포 (초기엔 개발자 모드 로컬 로드)
- ❌ 수익화 (광고/구독) — 포트폴리오 우선

## 12. 개발 단계

| 단계 | 내용 | 완료 기준 |
|---|---|---|
| 1 | 확장 뼈대 (MV3) + Readability 본문 추출 | 사이드패널에 현재 글 본문이 뜬다 |
| 2 | Claude adapter 1개 + **공통 뼈대 + [적용 모드] 조각** → 질문 JSON 렌더 | 버튼 누르면 질문+힌트가 나온다 |
| 3 | GPT / Gemini adapter 추가 (전략 패턴) | 3사 전환이 동작한다 |
| 4 | 설정 화면 (provider 선택, 키 입력, 프롬프트 편집) | 재빌드 없이 프롬프트 수정 가능 |
| 5 | 힌트 계단 UX + 성찰 마무리 다듬기 | 힌트가 하나씩 열리고 흐름이 매끄럽다 |
| 6 | **모드 조각 3개 추가 (검증·논증·소화) + 모드 드롭다운** | 일반 글에서도 적절한 질문이 나온다 |

> **모드는 한꺼번에 만들지 않는다.** 2단계에서 [적용 모드]만으로 핵심 루프를 검증한 뒤, 6단계에서 나머지 3조각을 붙인다.

> 먼저 **개발자 모드 로컬 로드로 본인이 직접 사용.** 3주 매일 써보고 유용하면 그때 다듬는다. (실사용이 유일하게 믿을 수 있는 수요 신호.)

## 13. 포트폴리오 관점 어필 포인트

이 프로젝트가 증명하는 역량:

- **멀티 provider 추상화** — 전략 패턴으로 3사 API 차이를 격리
- **브라우저 직접 호출 + 보안 트레이드오프 이해** — CORS, MV3 host_permissions, 키 저장 위치 선택
- **프롬프트 엔지니어링** — "정답을 주지 않는" 제약을 프롬프트로 강제하는 설계
- **제품 의사결정** — 왜 답을 가리지 않는가, 왜 자동 팝업을 금지하는가 등 명확한 근거 기반 설계
- **인지과학 기반 UX** — 이해 착각·전이·회수 개념을 실제 인터랙션으로 구현
- **확장 가능한 프롬프트 설계** — 장르(무한) 대신 독자 목적 4모드로 수렴, "뼈대+조각"으로 안전 규칙 단일화·모드 추가 저비용화

> 설계 결정의 "왜"를 회고 글로 남기면 코드보다 더 강한 포트폴리오가 된다.
