# 📖 ReCheck — 기획서

> **작업명(가칭): ReCheck** *"읽었다"와 "이해했다" 사이를 다시 확인시키는 크롬 확장*
> 문서 버전: v0.3 · 목적: 개인 학습용 + 포트폴리오
> v0.2 변경: 기술 글 한정 → **범용 4모드**(적용/검증/논증/소화) 구조로 확장. 프롬프트를 "공통 뼈대 + 모드 조각"으로 재편.
> v0.3 변경(구현 반영):
> - **문제집형 전환**: "원문 밖 전이 질문" → **"원문 특정 문단에 근거한 질문"**. 각 질문에 `anchor`(원문 그대로 복사한 구절)를 붙여, "원문에서 확인" 버튼으로 그 문단으로 스크롤·하이라이트 이동.
> - **UI 다국어(8개 언어)**: 영어 기본. 선택 언어로 UI뿐 아니라 **AI 질문·힌트 출력 언어**도 통일.
> - **드래그 선택 분석 + 12,000자 게이트**: 원하는 부분만 드래그해 분석. 선택 없이 전체 본문이 12,000자를 넘으면 통째 전송 불가 → 드래그 유도.
> - **프롬프트 내부 고정**: 사용자 편집(외부화) 폐기. 프롬프트는 코드 내부에 고정(영업 자산).
> - **모드는 자동 판별 고정**(수동 드롭다운 없음).

---

## 1. 한 줄 정의

읽은 글에 대해 사용자가 **스스로 자신의 이해를 다시 돌아보게** 만드는 크롬 확장. AI는 정답을 주지 않는다. **질문과 힌트만** 던지고, 답에 도달하는 것은 사용자의 몫이다.

---

## 2. 문제 정의 — 왜 만드는가

글이 매끄럽게 읽히면 뇌는 "안다"고 착각한다(**이해 착각, illusion of comprehension**). 하지만 막상 설명하거나 응용하거나 비판하려 하면 막힌다. 이 간극은 기술 문서뿐 아니라 뉴스·칼럼·에세이 등 **모든 종류의 글**에서 나타난다. 다만 글의 종류에 따라 "제대로 이해했는지"를 확인하는 방식은 달라야 한다 (→ 5장 4모드).

기존 도구(ChatGPT, Claude 등 챗봇)는 물어보면 **답을 준다.** 답을 받는 순간 사용자는 "그렇구나" 하고 닫으며, 이해 착각은 그대로 남는다. 회수(retrieval)가 일어나지 않으므로 학습으로 남지 않는다.

**ReCheck는 정답을 주지 않음으로써 이 문제를 푼다.**

---

## 3. 핵심 차별점 — 챗봇과 무엇이 다른가

| | 챗봇 (GPT/Claude) | ReCheck |
|---|---|---|
| AI의 역할 | 답을 준다 | 질문·힌트만 던진다 |
| 사용자 행동 | 읽고 수용 (수동) | 회수·적용 (능동) |
| 이해 착각 | 강화됨 | 파괴됨 |
| 최종 종착지 | AI의 답 | **원문으로 복귀** |
| 오개념 위험 | AI가 틀리면 주입됨 | **AI가 사실을 단정하지 않아 0** |

> **설계 대원칙**: AI는 "심판"이 아니라 "사다리"다. 정답 단어를 발화하지 않고, 사용자가 스스로 그 단어에 도달하도록 계단만 놓는다.

---

## 4. 설계 원칙 (변경 불가 핵심 규칙)

1. **정답 미제공** — AI는 어떤 질문에도 정답을 생성하지 않는다. 사실을 단정하는 문장을 출력하지 않는다.
2. **유저 호출 시에만 작동** — 자동 팝업 금지. 사용자가 사이드패널을 열고 버튼을 눌러야 질문이 나온다. (읽기 흐름을 끊는 개입은 제품을 죽인다.)
3. **원문 근거 질문 중심(문제집형)** — 글 내용을 그대로 되묻지도, 원문 밖을 상상하게도 하지 않는다. **원문의 특정 문단을 이해하면 답할 수 있는** 질문을 만든다. (v0.3: 기존 "전이 질문"에서 전환)
4. **힌트는 정답 직전에서 멈춘다** — 3단계로 점점 좁아지되, 마지막 단계도 정답 단어를 말하지 않는다.
5. **막힘의 종착지는 원문** — 각 질문에 `anchor`(근거 문단의 원문 구절)를 붙이고, **"원문에서 확인" 버튼**으로 그 문단으로 스크롤·하이라이트 이동시킨다. 답을 주는 게 아니라 "여기 다시 읽어라"로 돌려보낸다. (grounded 안전장치)
6. **성찰은 사용자 몫** — 답 작성 후 AI 평가 없이 "네가 쓴 답을 다시 읽어봐"로 마무리한다.

## 5. 질문 생성 전략 — 4모드

### 5.1 왜 "장르"가 아니라 "독자 목적"으로 나누는가

글 유형을 겉모습(장르: 뉴스·칼럼·논문·레시피·여행기…)으로 나누면 수십 개로 늘어나고 확장 불가능해진다. 대신 **"독자가 이 글을 왜 읽었는가"**로 나누면 4개로 수렴한다. 장르가 아무리 늘어도 새 글은 이 4개 중 하나에 떨어진다.

| 모드 | 핵심 질문 | 이 방식이 먹히는 글 | 독자 목적 |
|---|---|---|---|
| **① 적용 (Transfer)** | "이 개념이 성립하는 조건은 원문 어디에?" | 기술·개념·튜토리얼·방법론·레시피 | 써먹으려고 |
| **② 검증 (Scrutiny)** | "누가·무엇을·어떤 근거로? 근거 문단은?" | 뉴스·보도·발표 | 무슨 일인지 알려고 |
| **③ 논증 (Argument)** | "주장·근거는 원문 어느 문단에?" | 칼럼·사설·주장 글 | 누가 뭘 주장하는지 보려고 |
| **④ 소화 (Digest)** | "이 문단을 네 말로 재구성하면?" | 에세이·교양·서사 | 그냥 읽고 싶어서 |

### 5.2 모드별 질문 공식 (v0.3: 원문 근거형)

모든 모드는 **근거 문단(anchor)을 먼저 고르고 → 그 문단으로 답하는 질문**의 순서를 따른다(anchor-first). 이래야 질문과 하이라이트가 항상 같은 문단을 가리킨다. (실제 공식은 `src/prompt/modes.ts`)

**① 적용 — "핵심 개념 → 성립 조건 문단"**
개념이 성립하는 조건이 적힌 문단을 골라, 그 조건을 원문에서 짚게 한다.

**② 검증 — "사실 주장 → 근거·주체·조건 문단"**
사실 주장의 근거·주체·조건이 적힌 문단을 골라 확인하게 한다. **원문에 없는 "사회적 영향·함의·전망"은 묻지 않는다.**

**③ 논증 — "주장·근거 문단"**
글쓴이의 주장과 근거가 적힌 문단을 골라 짚게 한다.

**④ 소화 — "핵심 문단 재구성"**
핵심 문단을 골라 독자 자신의 말로 재구성하게 한다.

### 5.3 공통 원칙

네 모드 모두 **정답을 주지 않는다.** 질문 방식만 다를 뿐, "AI는 사다리일 뿐 심판이 아니다"라는 대원칙은 동일하다. 어떤 모드든 **원문의 중심 문단을 이해해야 답할 수 있는** 질문을 만들고, 그 문단으로 돌아가 다시 파악하게 한다(문제집형). 원문에 근거가 없는 추측·상상 질문은 금지한다.

## 6. 힌트 사다리 설계

각 질문마다 **3단 힌트 + anchor(원문에서 확인)**를 붙인다. 힌트는 가려진 상태로 나오고, 클릭하면 하나씩 열린다.

| 단계 | 성격 | 예시 (더티 체킹 질문) |
|---|---|---|
| 힌트 1 | 개념 방향만 | "더티 체킹은 '뭐가 바뀌었나'를 어떻게 알아낼까?" |
| 힌트 2 | 사고의 틀 좁힘 | "매번 정확히 추려 쿼리를 새로 짜기 vs 항상 같은 쿼리 재사용, 뭐가 쌀까?" |
| 힌트 3 | 정답이 적힌 위치로 (단정·용어 금지) | "원문에서 '이 기본 동작을 바꾸는 설정'을 언급한 문장을 다시 보세요." |
| anchor (원문에서 확인) | 최종 안전장치 | "📄 원문에서 확인" 버튼 → 근거 문단으로 스크롤 + 노란 하이라이트(3초) |

> 힌트는 세 단계 모두 **질문·포인터 형태**를 유지한다. 특히 힌트3도 정답 용어·명칭·수치를 말하지 않고 "원문 어디를 보라"로만 안내한다. (단정문 "~이다/~입니다" 금지)

- **anchor**: AI가 근거 문단에서 **원문 그대로 복사한 15~40자 구절**. 질문마다 "📄 원문에서 확인" 버튼이 있고, 클릭하면 content script가 그 구절을 페이지에서 찾아(`window.find`) 스크롤하고 CSS Custom Highlight로 노란 표시한다. 못 찾으면 "원문에서 못 찾음"으로 조용히 폴백.
- 화면에 실재하는 렌더 텍스트와 정제 본문의 공백·문단 경계 차이로 **일부 정확한 anchor도 못 찾을 수 있으나**, 폴백이 있어 안전하다.

> **힌트를 가려서 클릭으로 여는 UX는 '힌트'에만 적용한다. 정답에는 절대 적용하지 않는다.** (답을 가려두면 사용자는 무조건 먼저 열고, 그 순간 회수가 죽는다.)

## 7. 사용자 플로우

```
① 글 읽는 중 사이드패널 열기 (설정에서 언어·provider·키. 언어 기본 영어)
② (선택) 원하는 부분을 드래그 — 그 부분만 분석. 안 하면 본문 전체.
   패널에 선택/본문 글자수 실시간 표시. 12,000자 초과면 드래그 유도.
③ "이 글, 다시 짚어보기" 버튼 클릭
④ 본문(또는 선택분) 추출 → 선택한 provider API 호출 (1회)
   (이 1회 안에서 모드 자동 판별 + 근거형 질문 생성이 함께 이뤄짐)
⑤ 판별된 모드 표시 + 질문 2개 (각 3단 힌트 가려진 상태 + "원문에서 확인" 버튼)
⑥ 사용자가 답 작성 (막히면 힌트를 하나씩, 또는 "원문에서 확인"으로 근거 문단으로 이동)
⑦ "네가 방금 쓴 답을 다시 읽어봐" → 성찰 마무리
⑧ 끝 (저장 없음)
```

AI가 등장하는 지점은 **④ 질문 생성 딱 한 번.** 이후는 전부 사용자 혼자. 질문·힌트는 **선택한 UI 언어**로 나온다(글 본문 언어와 무관).

## 8. 기술 아키텍처

### 8.1 전체 구조 (서버 없음)

```
[크롬 확장 — Manifest V3]
 ├─ content script : 본문 추출(EXTRACT) · 선택 글자수 실시간(SELECTION_LENGTH)
 │                   · 본문 길이 조회(ARTICLE_INFO) · anchor 스크롤·하이라이트(SCROLL_TO)
 ├─ side panel (UI) : 언어·설정 · 질문/힌트 렌더 + provider API 직접 fetch + "원문에서 확인"
 ├─ service worker  : 아이콘 클릭 시 사이드패널 오픈 설정
 └─ chrome.storage.local : provider · 모델 · API 키 · UI 언어 저장
```

> API 호출은 service worker가 아니라 **사이드패널(확장 페이지)에서 직접 fetch**한다. MV3에서 확장 페이지의 fetch도 `host_permissions` 등록 도메인에 한해 CORS가 열리므로, 별도 메시지 중계 없이 UI에서 바로 호출하는 편이 단순하다. service worker는 패널 오픈 설정만 담당한다.

> **본문 추출 로직**: Readability 우선. 단 namu.wiki처럼 표·접기박스로 문단이 쪼개진 페이지에선 Readability가 본문을 크게 놓치므로, 컨테이너 렌더 텍스트가 훨씬 길면 그걸 본문으로 쓴다(하이브리드). 파악용 글자수와 실제 전송이 같은 로직(`extractBody`)을 공유한다.

> **content script 주입**: 매니페스트 선언형 주입 + 패널이 열릴 때 `chrome.scripting`으로 활성 탭에 직접 주입(이미 열려 있던 탭도 새로고침 없이 동작). 중복 주입 가드로 리스너 1회만 등록.

**서버 0 · 고정비 0.** 각 사용자가 자기 API 키를 넣고(BYOK), 자기 토큰 비용만 부담. 개인용·포트폴리오 목적에 최적.

### 8.2 멀티 Provider 추상화 (전략 패턴)

세 provider(GPT / Gemini / Claude)를 선택 가능. 차이를 adapter 한 곳에 가두고, 나머지 코드는 provider를 모른다. 각 adapter는 응답을 **"질문+힌트 JSON 하나"로 정규화**해서 반환한다.

| Provider | 엔드포인트 | 키 위치 | 특이사항 | 응답 경로 |
|---|---|---|---|---|
| Claude | `/v1/messages` | `x-api-key` 헤더 | `anthropic-dangerous-direct-browser-access: true` 헤더 필수 | `content[0].text` |
| GPT | `/v1/chat/completions` | `Authorization: Bearer` | 없음 | `choices[0].message.content` |
| Gemini | `/v1beta/models/*:generateContent` | URL `?key=` | 키를 URL에 노출 (로그 금지) | `candidates[0].content.parts[0].text` |

### 8.3 manifest.json 필수 설정

MV3 확장은 CORS 규칙이 웹페이지와 다르다. `host_permissions`에 세 API 도메인을 등록해야 사이드패널에서 외부 호출이 가능하다.

```json
"host_permissions": [
  "https://api.anthropic.com/*",
  "https://api.openai.com/*",
  "https://generativelanguage.googleapis.com/*"
],
"permissions": ["sidePanel", "tabs", "storage", "scripting", "activeTab"],
"content_scripts": [{ "matches": ["http://*/*", "https://*/*"], "js": ["src/content/extract.ts"] }]
```

> `scripting`+`activeTab`은 이미 열려 있던 탭에 content script를 직접 주입해 새로고침 없이 동작시키기 위함이다.

### 8.4 기술 스택

| 영역 | 선택 | 이유 |
|---|---|---|
| 플랫폼 | Chrome Extension **Manifest V3** | 크롬 확장 현행 표준 |
| UI 표면 | **Chrome Side Panel API** | 글과 나란히 두고 답을 쓸 수 있음 (팝업 ❌) |
| 프레임워크 | **Vue 3 + Vite** (+ `@crxjs/vite-plugin`) | 본인 강점 활용 + 포트폴리오 |
| 언어 | TypeScript | provider adapter 타입 안전성 |
| 본문 추출 | **`@mozilla/readability`** | 광고·사이드바 제거하고 본문만 추출 |
| 저장소 | **`chrome.storage.local`** | provider 선택값·API 키·프롬프트 저장 (서버 불필요) |
| AI 호출 | 각 provider REST API **직접 fetch** | 서버·프록시 없음 (BYOK) |
| 백엔드 | **없음** | 유지비용 0의 핵심 |

> **프레임워크 대안**: 극단적 단순함을 원하면 **Vanilla JS + HTML/CSS**로 빌드 스텝 없이 가는 것도 가능(의존성·유지보수 최소). 다만 Vue 3는 본인 주력 스택이라 포트폴리오 가독성·개발 속도 면에서 유리. 확장 자체가 작으므로 어느 쪽도 무겁지 않음. **권장: Vue 3 + Vite** (빌드 스텝은 '실행 비용'이 아니라 '개발 편의'이므로 유지비용과 무관).

### 8.5 유지비용 (0을 목표)

| 항목 | 비용 | 비고 |
|---|---|---|
| 서버/호스팅 | **0원** | 백엔드 없음. 확장은 정적 파일로 로컬 로드 또는 스토어 배포 |
| DB | **0원** | 저장은 `chrome.storage.local` (사용자 기기) |
| 도메인 | **0원** | 필요 없음 |
| AI 토큰 | **각 사용자 부담 (BYOK)** | 본인이 쓰는 만큼만 각 provider에 과금 |
| 스토어 등록 | 초기 0원 | 개발자 모드 로컬 로드. (웹스토어 배포 시 1회성 $5 등록비 — 나중 선택) |

> **유지비용을 0으로 만드는 단 하나의 설계 결정 = "서버를 두지 않는다".** 서버를 두는 순간 고정비·결제·남용 방어가 딸려온다. BYOK는 수익화를 포기하는 대신 이 비용 구조를 성립시킨다. 포트폴리오 목적에는 이 교환이 옳다.

## 9. 프롬프트 설계 (내부 고정)

**프롬프트는 이 제품의 90%다.** v0.3에서 **사용자 편집(외부화)을 폐기**하고 코드 내부에 고정했다 — 프롬프트가 제품의 핵심 자산이라 노출·수정 대상이 아니기 때문. 여전히 "공통 뼈대 + 모드 조각" 구조라 개발자는 재빌드로 튜닝한다.

### 9.1 내부 고정 방식 (구 "외부화" 폐기)

- 시스템 프롬프트는 코드(`src/prompt/skeleton.ts`, `modes.ts`)에 고정. 설정 화면에 프롬프트 편집 UI 없음.
- 안전 규칙(정답 금지 등)은 오직 **공통 뼈대 한 곳**에만 존재.
- ⚠️ 한계: 프롬프트 문자열은 빌드된 JS 번들에 평문으로 들어가므로, 확장을 뜯어보면 읽을 수 있다(클라이언트 실행 구조의 원리적 한계). 완전한 비밀 유지가 필요하면 서버 프록시로 옮겨야 하나 "서버 0" 설계와 상충 → 개인용/포트폴리오에선 감수.

### 9.2 출력 JSON 스키마 (확정 초안)

```json
{
  "mode": "적용 | 검증 | 논증 | 소화",
  "questions": [
    {
      "question": "질문 텍스트 (판별된 모드의 공식에 따름)",
      "hints": [
        "힌트1: 방향만",
        "힌트2: 사고의 틀 좁힘",
        "힌트3: 정답 직전 (정답 단어는 말하지 않음)"
      ],
      "anchor": "근거 문단에서 원문 그대로 복사한 15~40자 구절"
    }
  ]
}
```

- `mode`: 모델이 자동 판별한 글 유형(4모드 중 하나). UI에 표시. (수동 드롭다운은 없음)
- `questions` 배열 길이는 항상 2.
- `hints` 배열 길이는 항상 3.
- `anchor`(v0.3, 구 `source_pointer` 대체): 원문에서 한 글자도 안 바꾸고 연속 복사한 구절. "원문에서 확인" 이동의 검색 키. 요약·의역 금지.
- 질문·힌트·anchor는 모두 **선택한 UI 언어**로 출력한다(글 본문 언어와 무관).
- adapter들은 각 provider 응답을 이 스키마로 정규화한다(`parseQuestionSet`).

### 9.3 프롬프트 구조 — 공통 뼈대 1 + 모드 조각 4

프롬프트를 모드별로 통째로 4개 만들지 않는다. **공통 뼈대 하나**에 **판별된 모드의 조각 하나**를 끼워 조립한다. 이렇게 하면 안전 규칙(정답 금지 등)이 한 곳에만 있어 유지가 쉽고, 새 모드 추가도 조각 하나만 쓰면 된다.

```
최종 프롬프트 = 공통 뼈대  +  [모드 판별]  +  선택된 모드 조각
```

#### 9.3.1 공통 뼈대 (코드 고정 — 항상 동일)

> 실제 파일: `src/prompt/skeleton.ts`. `{{MODE_SECTION}}`(모드 판별+선택 모드 공식)과 `{{LANGUAGE}}`(출력 언어)는 `build.ts`의 `buildSystemPrompt(mode, language)`가 조립 시 주입한다. 아래는 요지.

```text
너는 사용자가 방금 읽은 글에 대해 "진짜 이해했는지" 스스로 돌아보게
만드는 질문 생성기다. 너의 역할은 오직 질문과 힌트를 던지는 것이며,
너는 절대로 답이나 설명, 사실 주장을 제공하지 않는다.

# 입력
사용자가 방금 읽은 글의 본문 텍스트가 주어진다.

{{MODE_SECTION}}   ← build.ts가 "모드 자동 판별 + 4개 공식" 또는 특정 모드 공식을 주입

## 힌트 (각 질문마다 정확히 3단계)
- 힌트1: 무엇을 떠올려야 하는지 "방향"만 가리킨다.
- 힌트2: 사고의 틀을 한 번 더 좁힌다.
- 힌트3: 정답 바로 앞까지 데려간다. 단, 정답에 해당하는
        핵심 단어·용어·이름은 절대 직접 말하지 않는다.

## 질문 만드는 절차 (각 질문마다 — anchor-first, 반드시 이 순서)
1. 먼저 "답의 근거가 담긴 문단"을 원문에서 고른다 → 그 문단의 15~40자 구절을 그대로 복사해 anchor.
2. 그 anchor 문단을 읽으면 답할 수 있는 질문을 쓴다. 질문과 anchor는 반드시 같은 문단을 가리킨다.
3. 원문에 답의 근거가 없는 것(사회적 영향·함의·전망 등)은 원문에 있는 척 묻지 않는다.

# 절대 규칙 (위반 시 실패)
- 정답을 말하지 마라. 어떤 사실도 단정하지 마라.
- 정답에 해당하는 핵심 용어를 힌트에 직접 쓰지 마라.
- 글을 요약하지 마라.
- 질문이 묻는 대상과 anchor 문단이 서로 다른 얘기면 실패다.
- 원문에 근거 없는 추측·상상·함의 질문 금지. 글의 중심 문단을 고른다.
- 질문 정확히 2개, 각 힌트 정확히 3개.
- 모든 질문·힌트·anchor를 반드시 {{LANGUAGE}}(으)로 작성하라.
  글 본문의 언어와 다르더라도 무조건 {{LANGUAGE}}로 쓴다.

# 출력 형식 (JSON만, 코드펜스·설명 금지)
{
  "mode": "적용 | 검증 | 논증 | 소화",
  "questions": [
    { "question": "...", "hints": ["...","...","..."], "anchor": "..." },
    { "question": "...", "hints": ["...","...","..."], "anchor": "..." }
  ]
}
```

#### 9.3.2 모드 조각 ([모드 공식] 자리에 끼워짐)

```text
[적용 모드 공식]
1. 핵심 개념 하나와, 그 개념이 성립하는 "조건"이 적힌 문단을 원문에서 고른다 → anchor.
2. 그 문단을 읽으면 답할 수 있는 질문을 만든다.
   (예: 이 개념이 성립하려면 원문은 어떤 조건을 전제하는가?)

[검증 모드 공식]
1. 핵심 "사실 주장" 하나와, 그 주장의 근거·주체·조건이 적힌 문단을 원문에서 고른다 → anchor.
2. 그 문단을 읽으면 답할 수 있는 질문을 만든다.
   (예: 누가, 무엇을, 어떤 근거로 판단·주장했는가?)
   원문에 적히지 않은 "사회적 영향·함의·전망"은 절대 묻지 않는다.

[논증 모드 공식]
1. 글쓴이의 핵심 "주장"과 그 "근거"가 적힌 문단을 원문에서 고른다 → anchor.
2. 그 문단을 읽으면 답할 수 있는 질문을 만든다. (예: 글쓴이는 무엇을 어떤 근거로 주장하는가?)

[소화 모드 공식]
1. 핵심 문단 하나를 원문에서 고른다 → anchor.
2. 그 문단을 독자 "자신의 말"로 재구성하게 하는 질문을 만든다.
```

> **v0.3 전환**: 기존 공식은 원문 밖(전이·반론·연결)으로 나가는 질문이었으나, 이제 모두 **원문 특정 문단에 근거**하고 그 문단을 `anchor`로 지목한다. "글 밖 상상"에서 "원문으로 돌아가 확인"으로 무게중심 이동.

### 9.4 구현 단계별 프롬프트 적용

MVP에서는 **[적용 모드]부터 하나만** 붙여 완성한다. 4모드를 동시에 튜닝하면 어느 모드가 왜 별로인지 추적이 안 된다. 적용 모드로 접근이 통하는 것을 확인한 뒤, 검증→논증→소화 조각을 하나씩 붙인다. (뼈대+조각 구조라 조각 추가 비용이 싸다.)

### 9.5 프롬프트 튜닝 시 관찰할 것

실제로 돌려보며 다음 실패 모드를 관찰하고 문구를 조정한다:

- **정답 누설** — 힌트3에서 핵심 용어를 말해버리는가? (가장 흔한 실패)
- **모드 오판** — 글 유형을 엉뚱하게 판별하는가? (자동 판별 고정)
- **단정형 혼입** — "~이다" 같은 사실 주장이 섞이는가?
- **anchor 불일치** — anchor를 의역·요약해서 원문에 그대로 없는가? → "원문에서 확인"이 못 찾음. (원문 그대로 복사를 강하게 지시)
- **언어 이탈** — 선택 언어가 아닌 글 본문 언어로 답하는가?
- **JSON 오염** — 코드펜스·설명이 앞뒤에 붙는가? (특히 Gemini. `parseQuestionSet`에서 방어)

## 10. 보안 고려사항

- API 키는 **반드시 `chrome.storage.local`에 저장.** `localStorage` 금지 (페이지 JS·악성 스크립트가 접근 가능).
- `chrome.storage.local`도 암호화 저장은 아님 — 확장 컨텍스트에 격리될 뿐. 개인용 로컬 설치라 현실 위험은 낮으나 인지할 것.
- **키를 코드에 하드코딩 금지.** (스토어 배포 시 키 탈취 위험 — 애초에 이 구조는 스토어 배포용이 아님.)
- Gemini는 키가 URL에 실리므로 **로그·에러 메시지·네트워크 캡처에 키 노출 금지.**
- Claude의 `dangerous-direct-browser-access` 헤더 이름이 경고하듯, 브라우저 직접 호출은 키 노출을 감수하는 구조. 개인용에 한정.

## 13. 포트폴리오 관점 어필 포인트

이 프로젝트가 증명하는 역량:

- **멀티 provider 추상화** — 전략 패턴으로 3사 API 차이를 격리
- **브라우저 직접 호출 + 보안 트레이드오프 이해** — CORS, MV3 host_permissions, 키 저장 위치 선택
- **프롬프트 엔지니어링** — "정답을 주지 않는" 제약을 프롬프트로 강제하는 설계
- **제품 의사결정** — 왜 답을 가리지 않는가, 왜 자동 팝업을 금지하는가 등 명확한 근거 기반 설계
- **인지과학 기반 UX** — 이해 착각·전이·회수 개념을 실제 인터랙션으로 구현
- **확장 가능한 프롬프트 설계** — 장르(무한) 대신 독자 목적 4모드로 수렴, "뼈대+조각"으로 안전 규칙 단일화·모드 추가 저비용화

> 설계 결정의 "왜"를 회고 글로 남기면 코드보다 더 강한 포트폴리오가 된다.
