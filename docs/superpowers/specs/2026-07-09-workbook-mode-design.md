# ReCheck — 문제집형(근거+이동) 전환 설계

> 문서 버전: v1 · 날짜: 2026-07-09
> 목적: 질문 생성 철학을 "원문 밖 전이"에서 **"원문 근거 + 문단으로 돌아가 파악"**(독해 문제집형)으로 전환하고, 각 질문에서 원문 근거 문단으로 **스크롤·하이라이트 이동**하는 기능을 추가한다.

---

## 1. 배경 / 문제

현재 질문은 "글이 멈춘 다음 칸"을 묻는 **전이(transfer) 질문**이라 답이 원문에 없다. 순수하게 머리로만 생각하게 만들어 **재미가 떨어지고**, 원문과의 연결이 약하다.

목표 경험은 **문제집**이다:
- 자료(원문)를 읽는다.
- AI가 그 원문의 **특정 문단에 근거한** 문제를 낸다.
- 사용자는 힌트를 참고해 **그 문단으로 돌아가(스크롤·하이라이트) 다시 읽고** 답을 파악한다.
- 질문은 드래그한 내용/전체 내용의 **중심에서 벗어나지 않는다.**

AI는 여전히 **답을 직접 주지 않는다.** 원문 문단이 "답을 확인하는 곳"이다. (제품 핵심 = 사다리, 심판 아님)

---

## 2. 범위

### 포함
- 질문 생성 프롬프트를 **근거형**으로 재작성 (4모드 유지, 공식 수정, 핵심 규칙 추가).
- 질문 스키마에 `anchor`(원문 그대로 복사한 근거 구절) 추가. 기존 `source_pointer` **대체**.
- 각 질문 카드에 **"원문에서 확인" 버튼** → 근거 문단으로 스크롤 + 하이라이트.
- 8개 언어 UI 문구 추가.

### 제외 (Out of Scope)
- PDF 지원 (별도 트랙).
- 힌트마다 개별 앵커 (질문당 1개로 고정).
- AI가 정답 텍스트를 제공하는 것 (금지 유지).
- 다중 문단 하이라이트 (앵커 1개 = 한 곳).

---

## 3. 질문 생성 변경

### 3.1 모드 (4모드 유지, 공식 근거형 재작성)
자동 판별은 유지. 각 모드 공식을 "원문 근거"로 수정한다.

- **적용**: (기존 "조건이 깨지면?" → 글 밖) → **"핵심 개념이 성립하는 조건을 원문에서 찾아 짚게 하는 질문."**
- **검증**: 핵심 사실 주장과 그 함의를 **원문 문단에서 확인**하게 하는 질문.
- **논증**: 글쓴이의 주장·근거를 **원문에서 짚게** 하는 질문.
- **소화**: 핵심 문단을 **자신의 말로 재구성**하게 하되, 근거가 되는 문단을 지목.

### 3.2 뼈대(skeleton) 규칙 추가
- 질문은 반드시 **anchor로 지목한 문단을 이해하면 답할 수 있어야** 한다. (원문 밖 상상·추측 금지)
- 글의 **중심 내용**을 다루는 문단을 고른다. 지엽적·부차적 문단 금지.
- (유지) AI는 답·설명·사실 단정을 하지 않는다. 힌트는 정답 직전에서 멈춘다.
- (유지) 질문 2개, 힌트 3개, UI 선택 언어로 작성.

### 3.3 스키마 변경
`Question`에서 `source_pointer`를 제거하고 `anchor`를 추가한다.

```ts
export interface Question {
  question: string
  hints: string[]     // 3단계 (유지)
  anchor: string      // 근거 문단에서 "그대로 복사한" 15~40자 연속 구절
}
```

출력 JSON 형식(뼈대·build.ts의 안내)도 `source_pointer` → `anchor`로 교체하고,
"anchor는 원문에서 **한 글자도 바꾸지 말고** 연속으로 복사한 구절"임을 강하게 명시한다.

`parseQuestionSet`는 `anchor`를 문자열로 매핑(방어적 처리 유지).

## 4. 점프 UI + 하이라이트

### 4.1 UI (App.vue)
- 각 질문 카드에 **"📄 원문에서 확인"** 버튼을 **처음부터** 표시. 힌트(3단계)는 지금처럼 읽기 전용으로 유지.
- 클릭 → 활성 탭 content script에 `{ type: 'SCROLL_TO', text: question.anchor }` 전송.

### 4.2 하이라이트 (content script)
- 받은 `anchor` 구절을 페이지에서 찾는다.
  - 1차: 공백 정규화 후 텍스트 노드들을 순회해 매칭되는 `Range` 구성.
  - 못 찾으면 실패 응답.
- 찾으면: `range` 기준 `scrollIntoView({ behavior: 'smooth', block: 'center' })` + **CSS Custom Highlight API**로 노란 형광 표시. **약 3초 후 자동 해제.** (DOM 미변경 → 원본 안전)
- 성공/실패를 sendResponse로 회신.

### 4.3 폴백
- 못 찾으면 패널에서 해당 버튼에 **"원문에서 못 찾음"**을 잠깐 표시(에러 토스트 아님, 조용히).
- 앵커는 분석에 실제로 보낸 텍스트(드래그 선택분 또는 전체 본문)에서 뽑히므로 **화면에 실재**한다 → 대부분 매칭. 단, AI 의역·정제본과 렌더 텍스트 차이로 실패 가능(100% 아님).

## 5. i18n
추가 문구(8개 언어):
- `openInSource`: "원문에서 확인" (버튼)
- `notFoundInSource`: "원문에서 못 찾음" (폴백)

기존 키(`truncNote`/`articleTooLong` 등)는 그대로 유지(이번 변경과 무관). `source_pointer`는 UI에서 쓰지 않으므로 관련 표시 없음(신규 버튼으로 대체).

## 6. 영향 파일
- `src/prompt/modes.ts` — 4개 공식 근거형 재작성.
- `src/prompt/skeleton.ts` — 규칙 추가, 출력 형식 `source_pointer` → `anchor`.
- `src/providers/types.ts` — `Question.source_pointer` → `anchor`, `parseQuestionSet` 매핑.
- `src/content/extract.ts` — `SCROLL_TO` 처리 + 구절 검색·하이라이트.
- `src/sidepanel/App.vue` — 결과 카드에 "원문에서 확인" 버튼, 클릭 핸들러, 폴백 표시.
- `src/i18n.ts` — 문구 2개 × 8개 언어.

---

## 7. 성공 기준 (검증 가능)
1. 근거형 프롬프트로 생성 시, 각 질문이 **원문 특정 문단을 이해하면 답 가능**하고 `anchor`가 그 문단의 실제 구절이다.
2. "원문에서 확인" 클릭 → 해당 문단으로 **스크롤 + 노란 하이라이트**가 뜨고 약 3초 후 사라진다.
3. anchor를 페이지에서 못 찾는 경우 → **조용한 폴백 표시**, 크래시·콘솔 에러 없음.
4. 드래그 선택 분석·전체 분석 **양쪽 모두**에서 앵커 이동이 동작한다.
5. UI 언어를 바꾸면 버튼·폴백 문구도 번역된다.
6. `npm run build`(vue-tsc 타입체크 포함) 통과.
